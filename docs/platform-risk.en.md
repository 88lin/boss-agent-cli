# Platform Risk Boundaries

boss-agent-cli defaults to `assisted`: local assistance, read-only first, and user-triggered. Users may explicitly enable `research` for browser-protocol, anti-debugging, risk-control adaptation, and controlled collection research. The project does not control platform rules, account risk control, API drift, or third-party browser environments.

## 0. Default Low-Risk Mode

Default Low-Risk Assistance Mode blocks these sensitive workflows:

- automated greeting, batch greeting, applying / starting conversations, contact exchange
- recruiter candidate search, applications, online resumes, chat history, recent-message summaries
- recruiter replies, attachment-resume requests, phone / WeChat exchange requests

Complete those actions manually on the official website while in assisted mode. Research Mode exposes only capabilities declared by the policy registry and does not remove redaction, bounded-run, checkpoint, or stop-control requirements.

## 0.1 Explicit Research Mode

```bash
boss config set operating_mode research
boss schema --format native
```

Research Mode requires explicit activation, bounded pages/retries/runtime, checkpoint-before-resume on risk signals, an independent browser profile, redacted credentials and identifiers, auditable repository/commit/license/hash metadata for vendored hooks, and local retention/cleanup controls. A zero value must not silently mean an unlimited run.

## 1. Platform APIs can drift

The project depends on web pages, APIs, cookies, login state, and response shapes from platforms such as BOSS Zhipin and Zhaopin. These platforms can change fields, risk-control behavior, endpoint paths, or page flows at any time.

Treat these symptoms as possible platform drift:

- Previously working read-only commands suddenly return `NETWORK_ERROR`, `AUTH_EXPIRED`, `TOKEN_REFRESH_FAILED`, or malformed structures.
- `search` returns results but `detail` fails.
- `boss schema --format native` works, but live commands fail.
- Mock tests pass while real-account commands fail.

## 2. Login and Cookie boundaries

Login can use Cookie extraction, CDP, QR httpx, or browser fallback. The project reads and stores login state locally. Users should never submit cookies, tokens, phone numbers, WeChat IDs, names, company-private information, or real `security_id` values to the repository. Assisted login compatibility must not become a risk-control retry channel; Research Mode adapters follow section 0.1.

`boss status` checks local encrypted credentials and layered health by default without contacting the platform. Use `boss status --live` or `boss doctor --live-probe` only when you explicitly want a read-only online probe. A `wt2` cookie without `__zp_stoken__` is a partial login state; the stoken is generated by real page JavaScript, so users may run `boss login --cdp` with a Chrome CDP debugging port they started intentionally. CDP must not be used as a risk-control bypass channel.

Redact issue payloads:

```json
{
	"security_id": "<redacted>",
	"cookie": "<redacted>",
	"token": "<redacted>"
}
```

## 3. Request rate and account responsibility

The default request interval is controlled by `--delay`. Assisted Mode does not perform high-frequency scraping or bulk outreach. Research collection and write actions still require capability policy, bounded budgets, and explicit user actions.

## 4. Browser automation boundaries

patchright, CDP, local Chrome profiles, system keychains, browser extensions, and platform risk-control systems all affect login and access stability. Assisted Mode stops when the platform flags automation. Research Mode may use declared adapters to study that behavior, but must not switch channels or retry without bounds.

Windows app clients, visible browsers, CloakBrowser, RPA tools, and fingerprint browsers all follow the same operating-mode policy. Mainline may accept auditable Research Mode adapters, but assisted smoke tests must never run live risk-control research automatically.

Third-party implementations using stealth, response interception, auto-scroll collection, bulk extraction, or simulated real-user fingerprints may enter a Research Mode adapter only after repository, license, commit, hash, data-path, and stop-behavior review. They must not enter the assisted default path or real-account CI. Every new platform must first pass the [platform adapter research template](research/platforms/README.md) gate.

## 5. Smoke-test boundaries

Real-flow smoke tests must be explicitly configured and should not access real accounts from regular CI:

```bash
BOSS_SMOKE_DRY_RUN=1 uv run python scripts/smoke_p0.py
BOSS_SMOKE_PLATFORM=zhipin BOSS_SMOKE_QUERY=Golang BOSS_SMOKE_SECURITY_ID=<redacted> uv run python scripts/smoke_p0.py
```

`BOSS_SMOKE_DRY_RUN=1` verifies the plan only. It does not prove live platform availability.

## 6. Reporting security issues

If a report involves cookies, tokens, accounts, contact details, private resumes, company-private information, or exploitable automation bypasses, do not open a public issue. Use the private disclosure channels in [SECURITY.md](../SECURITY.md).
